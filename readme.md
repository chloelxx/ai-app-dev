实战项目体系（重点：agent 目录）
所有 AI 应用实践项目应放在 projects/agent/ 目录下，按从小到大、逐步递进的方式组织。
目录结构约定
projects/agent/
01-basic-agent/ 小型项目：单一工具 / 单一任务
02-rag-agent/ 基础 RAG + Agent
03-multi-agent-workflow/ 多智能体协作与工作流
04-enterprise-agent-service/ 企业级 Agent 服务
每个项目目录内应至少包含：
project-guide.md：项目目标、背景、需求说明、验收标准
src/：主要代码（建议用 FastAPI 作为服务外壳）
tests/：关键功能的测试用例
README.md：运行说明与项目总结
项目递进设计（从小到大）
01 Basic Agent：基础能力打底
目标：实现一个简单的工具增强聊天 Agent
要求：
集成一个主流大模型 API（如 OpenAI 或 Claude）
至少一个工具调用（如天气查询 / 计算器 / 文本翻译）
基本提示工程和会话管理
关注点：API 调用、提示设计、错误处理、代码结构
02 RAG Agent：知识库问答
目标：基于自建知识库的 RAG 问答 Agent
要求：
使用向量数据库（如 Chroma 或 Milvus）
实现文档解析、分块、嵌入与索引构建
问答流程：用户问题 → 检索相关文档片段 → Agent 结合上下文回答
关注点：RAG 架构、检索质量、上下文拼接策略
03 Multi-Agent Workflow：多智能体协作
目标：实现一个多步骤工作流，由多个 Agent 协作完成
要求：
至少 2–3 个角色分工明确的 Agent（如：规划 Agent、执行 Agent、审阅 Agent）
具备简单的任务分解和结果汇总机制
可以使用 LangChain / AutoGen / CrewAI 等框架之一
关注点：Agent 角色设计、消息路由、错误与重试策略
04 Enterprise Agent Service：企业级应用雏形
目标：面向企业场景的端到端 Agent 服务（如智能客服 / 知识助手）
要求：
使用 FastAPI 暴露 HTTP API 或 Web UI
集成 RAG、多 Agent 或工具链
容器化部署（Docker），配备日志和基本监控方案
关注点：架构设计、可维护性、性能与部署
